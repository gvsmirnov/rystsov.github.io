Visualization of distributed FSM over Paxos

The main result of the Paxos Made Simple paper by Leslie Lamport is an algorithm to transform any finite-state machine (FSM) into a fault-tolerant distributed system.

It sounds complicated but with the step-by-step vizualization and this post I'm going to demystify its complexity.

Let us start with FSM. It's a fancy name for a pretty wide class of applications. How wide is it? If you can model your computation as a pure function from a state and an event into a new state and you have the same limit on the size of each state/event then your computation (your application) is a FSM.

You can guess that almost any web application using three-tier architecture matches the description: the stateless servers play a role of the pure function while size of the database is limited by the size of HDD.

Now it should be clear why Paxos is important: it allows to convert almost any application into a distributed system to achieve more reliability than each node provides.

The paper was written in 2001 and it contains several breakthriught ideas:

  • Leslie Lamport demonstarated how to build linearizable distributed data structures:
    • a distributed write-once register
    • a distributed append-only log
  • He suggested to use distributed append-only log to store sequence of events leading a system to the current state instead of storing the state inself (later this idea was rediscovered as the event sourcing pattern).

In 2016 it looks natural to use logs for FSM (see ZooKeeper, Etcd, Raft). But logs aren't the first thing that I want to work with as an engineer because they force me to implement garbage collection and snapshots (nasty work, see Paxos Made Live paper by Google). Alse it's possible to imagine how other than logs distribibuted data structures could model FSM I wonder if they really do and why FSMs over them aren't known.

  • They do. For example TreodeDB used this approach. It's interesting that I haven't seen any academic paper dedicated to this idea, but I saw several industry systems built with it, so I practiced my math fu, described and proved it in the How Paxos works post.
  • IMHO the log-based solutions are more popular because they had time advantage. Until Vertical Paxos was published in 2009, log replication was the only consensus protocol capable of cluster membership change thous it was the only practical solution for highly available services. But now we have a couple algorithms to choose from, for example Vertical Paxos and Raft's joint consensus membership change are among them. Take a look on the Best of both worlds: Raft's joint consensus + Single Decree Paxos post if you're interested in understanding how dynamic membership works with a distributed variable.

Visualization

Now when we understood that there is a place for alternative distributed FSM let's discuss the visualization. We're going to vizualized the work of distributed FSM managing an intercontinental ballistic missile launch system.

FSM

Imagine that we want to model an intercontinental ballistic missile launch system with the following rules:

  1. The missile is launched when at least two generals signed off the launch.
  2. Until missile is launched a general can revoke their sign off

The rules define the possible state of the system; while the actual state is defined by the generals behaviour which is quite simple in our example:

  1. unsign the launch
  2. sign the launch
  3. goto 1
Distributed FSM

Distributed FSM has several endpoints with the execute service, but it doesn't matter which endpoint you choose for any execute call because the system behaves exectly as a system with several endpoints but leading to the same (non-distributed) FSM. The only difference that the distributed version runs on 2n+1 nodes and tolerates up to n failures.

Visualization

The vizualization consists of two code columns and the data side view. The first column contains code of a distributed Single Decree Paxos FSM engine (36 lines of code); while the second column is dedicated for the state transition functions and the generals behaviour (33 lines of code).

It worths noticing that the state transition functions weren't changed at all, so it extrimly easy to convert existing FSM into DFSM. By the way the actions in the example look like message handlers in the actor model, so FSM over SDP looks like the great way to store the state of the actors in Akka or Orleans.
